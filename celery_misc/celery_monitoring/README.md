# Celery Monitoring: Модуль мониторинга задач Celery с интеграцией в Django Admin

Компонент обеспечивает наблюдение за запущенными задачами Celery, ведет историю их выполнения и предоставляет функционал
поиска по историческим данным. Решение отличается простотой подключения и не требует развертывания дополнительной
инфраструктуры.

## Быстрый старт

1. Установите последнюю версию пакета или добавьте ее в requirements.txt

```
pip install celery-misc==1.0.1
```

2. Добавьте периодический запуск задачи delete_running_log_task, используя django-celery-beat. Задача будет очищать лог
   системы мониторинга и по умолчанию оставлять историю за последние сутки.

```python
from celery.schedules import crontab

CELERY_BEAT_SCHEDULE = {
    'Зачистка лога выполненных задач': {
        'task': 'celery_misc.celery_monitoring.tasks.delete_running_log_task',
        'schedule': crontab(minute='*/20'),  # every 20 min
    }
}
```

3. Добавьте нужные модули в настройках Django. django_json_widget нужен для красивого отображение json полей в Django
   AdminUI.

```python
INSTALLED_APPS = [
    ...,
    'celery_misc.celery_monitoring',
    'django_json_widget'
]
```

4. Вы прекрасны! Наслаждайтесь историей запусков ваших Celery задач в разделе "Мониторинг задач" в Django AdminUI.

## Основные возможности

1. Ведение истории запусков задач Celery. Отслеживание их статусов, продолжительности и параметров запуска.
2. Хранение истории за последние сутки и поиск по ней. Возможность настроить время хранения истории.
3. Работа с состоянием и результатом выполнения задач. Состояние выполнения позволит восстановить работу задачи с того,
   места, на котором она прервалась. Функции monitoring_utils.save_state и monitoring_utils.load_state
4. Возможность перезапуска задачи. Функция monitoring_utils.restart_task.
5. Возможность гарантированного прерывания длительных задач. Функции monitoring_utils.find_task, instance.revoke_task и
   Terminator.
6. Отслеживание количества попыток перезапуска задач.
7. Ведение общего репозитория с "белым" и "черным" списком видимости задач для системы мониторинга.
8. Возможность профилирования задач(в процессе реализации) в части CPU и ОЗУ.
9. Возможность динамической настройки задач в процессе выполнения из репозитория задач.

## Алгоритм определения доступности задачи для мониторинга

1. По умолчанию все задачи доступны для мониторинга. Если требуется настроить видимость конкретной задачи, ее необходимо
   добавить в раздел «Репозиторий задач».
2. После включения задачи в «Репозиторий задач» к ней применяются правила фильтрации, описанные ниже.
3. Внесение задачи в «Черный список задач» автоматически делает ее недоступной для системы мониторинга. Черный список
   имеет безусловный приоритет над белым списком.
4. Белый список активируется только при наличии хотя бы одной записи в разделе «Белый список задач». В этом случае
   доступными для мониторинга остаются лишь задачи, явно добавленные в белый список (при условии, что они не находятся в
   черном списке). Если белый список не пуст, все прочие зарегистрированные задачи становятся недоступными для
   мониторинга.
